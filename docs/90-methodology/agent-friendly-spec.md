# AI-Agent Friendly Spec: 큰 원칙

> 적용: 전 섹션 작성 원칙. 파이프라인 연계: `.claude/rules/docs-pipeline.md`. 출처: [[index|방법론 인덱스]].

## 목적
기획문서가 자동 구현 환경에서 흔들리지 않게. 구현 세부를 과도하게 지시하는 게 아니라,
**에이전트가 임의로 결정하면 안 되는 제품/기획/UX 판단을 문서에서 고정**한다.

## 원칙 1 — 문서가 제품 결정을 소유한다
반드시 문서에 포함: 한 문장 컨셉 · 핵심 재미 가설 · Core Loop 단계 · 주요 Entity와 시스템 책임 ·
상태 전이와 입력 정책 · 밸런스 수치와 의도 · 화면 레이어와 UI 우선순위 · 완료 기준 · 제외 기능.

## 원칙 2 — 암묵적 판단을 남기지 않는다
"적당히/자연스럽게/보기 좋게/캐주얼하게"를 관찰 가능한 기준으로 바꾼다.
| 나쁜 표현 | 좋은 표현 |
|---|---|
| 적당한 속도로 움직인다 | 1초에 화면 높이의 20%를 이동 |
| 보상을 충분히 준다 | 첫 30초 안에 최소 1회 보상 |
| UI를 보기 좋게 배치 | 재화=상단 좌, 타이머=상단 중앙, 목표=상단 우 |
| 난이도를 점점 올린다 | 30초마다 스폰 간격 10% 감소 |
| 합성하면 강해진다 | 레벨 1 상승 시 생산량 2배 |

## 원칙 3 — Scope Fence를 둔다 (포함/제외 명시)
기본 제외 예: 로그인·계정·서버 저장 / 결제·광고·IAP / 복잡한 메타 성장 / 온라인 랭킹 /
고품질 아트 제작 / 사운드 에셋 필수 의존 / 다국어 / 장기 저장 / 튜토리얼 모든 예외 처리 /
모바일 앱 패키징. (프로토타입 목적상 일부는 포함으로 조정 가능.)

## 원칙 4 — Must / Should / May 구분
| 키워드 | 의미 |
|---|---|
| Must | 완성에 반드시 필요 |
| Should | 가능하면 포함, Core Loop보다 우선 안 함 |
| May | 시간 남을 때 선택 |
| Must Not | 구현/추가 금지 |

## 원칙 5 — 단일 출처를 만든다 (SSoT)
| 항목 | 단일 출처 |
|---|---|
| 밸런스 수치 | Balance Table / Config |
| UI 문구 | UI Text Table / Config |
| Entity 기본값 | Entity Config |
| 상태 전이 | State Machine Table |
| 이벤트 정의 | Event Catalog |
| 완료 기준 | Acceptance Checklist |

동일 정보가 여러 곳에 중복되면 에이전트가 어느 값을 따를지 판단해야 하므로 중복을 줄인다.

## 원칙 6 — 판단 우선순위 지정 (충돌 시)
1. Core Loop 플레이 가능성 → 2. 입력·피드백 명확성 → 3. 밸런스 검증 가능성 →
4. 상태 전이·재시작 안정성 → 5. 코드 구조 단순성 → 6. 시각 완성도 → 7. 부가 연출.
즉 아트/구조를 위해 플레이 가능한 결과물이 늦어지면 안 된다.

## 원칙 7 — 예외의 기본값을 정한다
| 예외 | 기본 정책 |
|---|---|
| 재화 부족 상태 구매 클릭 | 버튼 비활성 또는 부족 피드백 |
| 팝업 중 월드 클릭 | 무시 |
| 세션 종료 후 입력 | 결과 화면 버튼만 허용 |
| 화면 밖 Entity | 제거 또는 경계 안으로 제한 |
| 중복 이벤트 | 같은 프레임 내 동일 이벤트 중복 처리 금지 |
| 설정값 누락 | 임의 보정 금지, 기본값 테이블 기준 |

## 완료 기준
- 에이전트가 임의로 장르·수치·UI·상태를 결정할 여지가 적음
- Core Loop·State Machine·Balance·Event·Layer·Acceptance가 서로 연결됨
- 기능 우선순위·제외 범위 명확
- 모호한 표현보다 관찰 가능한 기준이 많음
- 문서만 읽어도 "무엇을 만들면 완료인지" 판단 가능
