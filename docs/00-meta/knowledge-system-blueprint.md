# Knowledge System Blueprint

`docs/`는 이 프로젝트의 **단일 진실 원천(SSoT)** 이자, 동시에 제출 산출물인
**에이전트 실행용 기획문서(GDD + build-spec)** 다. 게임을 만드는 과정에서 훅으로 갱신되어
항상 최신 정본을 유지한다.

## 무엇을 위한 문서인가 (과제 맥락)

베이글코드 [신작팀] 게임 기획 PD 과제. 목표는 **AI 코딩 에이전트(Claude Code 등)에게
전달했을 때, Core Loop를 플레이할 수 있는 HTML5 캐주얼 게임 프로토타입을 만들어낼 수 있는
기획문서**를 작성하는 것이다.

- 장르: **Merge(합성)** — 세부 메카닉 미정.
- 구현 대상: HTML5 플레이 가능한 프로토타입 (pixi.js / Canvas / three.js — **스택 미정**).
- 제출: `.md` 로 구성된 기획문서(zip). 선택적으로 에이전트 실행 결과물(`game/`) 동봉.
- 마감: 2026-06-28 23:59.

## 산출물 3종 (이 레포가 담는 것)

1. **기획문서(정본)** — `docs/` 의 번호 섹션들. 과제 제출 본문.
2. **게임 프로젝트 파일** — `game/` (HTML5 프로토타입; 스택 결정 후 생성).
3. **파이프라인 인프라 + 지시 원본** — `docs/00-meta/` (훅 규칙, 템플릿, **input-log**).

## 파이프라인 (log → reconcile → work) — 변경 없음

게임을 만드는 동안 문서가 자동으로 최신 정본을 유지하게 하는 핵심 장치.

1. **log** — 모든 사용자 지시를 `00-meta/input-log/<날짜>.md` 에 **그대로** 기록
   (`UserPromptSubmit` 훅, 무조건).
2. **reconcile** — 지시가 기존 문서와 다르면 **해당 섹션 문서를 먼저 수정**한다.
3. **work** — 수정된 문서를 근거로 작업(설계/구현)한다. `Stop` 훅이 누락을 잡는다.

전체 규칙: `.claude/rules/docs-pipeline.md`.

## 핵심 규칙

1. **input-log는 신성하다.** 손으로 편집 금지 — 지시(요청)의 원본.
2. **요청과 정본을 분리한다.** 요청은 `input-log/`, 결정/설계는 번호 섹션 문서.
3. 한 문서는 한 주제(컨셉/루프/시스템/결정)에 집중한다.
4. **새 지시가 문서와 충돌하면 문서를 먼저 고친 뒤 작업한다.**
5. **미정인 것을 단정하지 않는다.** 장르 세부·스택 등 미정 → `status: draft` + 열린 질문.
6. **에이전트 실행성**이 최우선 기준이다. 모든 섹션은 "이대로 에이전트에게 주면 실행되는가"로 검증.
