# 핵심 원칙

> 적용: 전 섹션. 기준 출처: [[index|방법론 인덱스]].

이 문서들은 상세 기획서가 아니라 **이후 작성할 기획서·구현 지시서에 적용할 구조적 원칙**이다.
함수 단위 마이크로 지시보다 다음 큰 결정을 고정하는 데 초점을 둔다: 단일 출처 데이터 / 오브젝트
분해 단위 / 흐름·UX 상태 전이 / 프레임 처리 순서·시간 기준 / 시스템 의존성 / 레이어·입력
우선순위 / 완성 검증 기준.

## 1. 결정은 문서가 한다
에이전트가 스스로 기획 판단을 내리면 검증 목적이 흐려진다. 다음은 **반드시 문서에서 결정**한다.
- Core Loop 단계와 반복 조건 · 플레이어 주요 행동 · 보상 지급 조건
- 실패/압박/성장/재시작 조건 · 주요 수치와 밸런스 의도
- UI 표시 우선순위 · 입력 가능/불가 상태 · 프로토타입 포함/제외 범위

## 2. 움직이는 값은 데이터로 고정한다
밸런스, 스폰 간격, 비용, 보상량, 세션 길이, 레벨 조건 등 변경 가능성이 높은 값은 코드에
흩어지지 않게 한다. 조정 가능한 값은 **별도 밸런스 테이블/설정 객체**에서 관리. → [[data-driven]]

## 3. 흐름은 상태로 고정한다
진행 흐름은 상태머신으로 정의한다. `isPlaying`/`isPaused`/`isGameOver`/`isTutorial` 같은
**Boolean 플래그 조합으로 흐름을 암묵화하지 않는다**. 상태별 허용 입력·시스템 실행·UI 표시를
명시. → [[state-machine]]

## 4. 시스템 간 직접 결합을 줄인다
전투·생산·병합·UI·이펙트·사운드·진행도 시스템이 서로를 직접 호출하지 않게 한다. 주요 변화는
이벤트로 발행하고 필요한 시스템이 구독해 반응. → [[event-driven]]

## 5. 렌더링과 시뮬레이션을 분리한다
그리는 일과 상태를 바꾸는 일을 섞지 않는다. **상태 업데이트 순서**와 **렌더 레이어 순서**를
별도로 정의. → [[game-loop]] · [[layered-rendering]]

## 6. 완료 기준은 실행 가능한 체크리스트로 쓴다
완료 여부는 주관적 설명이 아니라 실행 가능한 체크리스트로 판단한다. "재미있게 느껴진다"가
아니라 **"3분 세션 안에 핵심 루프를 5회 이상 반복할 수 있다"** 처럼 확인 가능한 문장으로.
→ [[acceptance-test]]
